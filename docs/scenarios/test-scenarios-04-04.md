## 1. 인증 (Authentication)

### 1.1 로그인 (`/login`)

#### 정상 시나리오

- [ ] **SC-001**: 이메일/비밀번호 정상 로그인
  - 조건: 가입 완료 및 이메일 인증 완료된 계정
  - 동작: 유효한 이메일, 비밀번호 입력 후 "로그인" 클릭
  - 기대: accessToken 발급, 워크스페이스가 1개면 해당 `/{ws}/doc`로 자동 이동, 2개 이상이면 `/workspaces`로 이동 (FR-003)

- [ ] **SC-002**: "로그인 상태 유지" 체크 후 로그인
  - 조건: 유효한 계정
  - 동작: rememberMe 체크박스 활성화 후 로그인
  - 기대: Refresh Token이 HttpOnly Cookie로 설정됨, 브라우저 재시작 후에도 세션 유지

- [ ] **SC-003**: 로그인 폼 Enter 키 제출
  - 조건: 이메일, 비밀번호 필드 입력 완료
  - 동작: 비밀번호 필드에서 Enter 키 입력
  - 기대: 폼이 정상 제출되어 로그인 처리

#### 에러 시나리오

- [ ] **SC-004**: 빈 필드로 로그인 시도
  - 조건: 이메일 또는 비밀번호가 빈 상태
  - 동작: "로그인" 클릭
  - 기대: "이메일과 비밀번호를 입력해주세요." 에러 메시지, 빈 필드에 빨간 border

- [ ] **SC-005**: 잘못된 비밀번호로 로그인
  - 조건: 유효한 이메일, 틀린 비밀번호
  - 동작: 로그인 시도
  - 기대: `INVALID_CREDENTIALS` 에러, "이메일 또는 비밀번호가 올바르지 않습니다." 표시

- [ ] **SC-006**: 미인증 이메일로 로그인
  - 조건: 이메일 인증 미완료 계정
  - 동작: 로그인 시도
  - 기대: `EMAIL_NOT_VERIFIED` 에러, "이메일 인증이 완료되지 않았습니다. 메일함을 확인해주세요." 표시

- [ ] **SC-007**: 계정 잠금 (5회 연속 실패)
  - 조건: 동일 계정에 4회 로그인 실패 상태
  - 동작: 5번째 실패 시도
  - 기대: `ACCOUNT_LOCKED` 에러, 카운트다운 타이머 (`MM:SS` 형식) 표시, 잠금 해제까지 로그인 버튼 비활성화

- [ ] **SC-008**: 계정 잠금 카운트다운 완료
  - 조건: 계정 잠금 상태에서 카운트다운 0초 도달
  - 동작: 카운트다운 종료 대기
  - 기대: 에러 메시지 자동 제거, 로그인 버튼 재활성화

- [ ] **SC-009**: Rate Limit 초과
  - 조건: IP 기준 15분 내 10회 초과 요청
  - 동작: 로그인 시도
  - 기대: `RATE_LIMITED` 에러, 경고(amber) 스타일로 "로그인 시도가 너무 많습니다." 표시

- [ ] **SC-010**: 네트워크 오류
  - 조건: API 서버 연결 불가 상태
  - 동작: 로그인 시도
  - 기대: "네트워크 오류가 발생했습니다. 다시 시도해주세요." 에러 표시

#### 엣지 케이스

- [ ] **SC-011**: 로그인 중 중복 제출 방지
  - 조건: 로그인 요청 진행 중
  - 동작: "로그인" 버튼 재클릭 시도
  - 기대: 버튼이 "로그인 중..." 텍스트와 함께 disabled 상태, 중복 요청 발생하지 않음

- [ ] **SC-012**: Access Token 만료 시 자동 갱신
  - 조건: accessToken 만료, refreshToken 유효
  - 동작: API 요청 발생
  - 기대: `/auth/refresh` 호출하여 새 accessToken 발급, 원래 요청 재시도 성공

- [ ] **SC-013**: Refresh Token도 만료 시 로그아웃
  - 조건: accessToken, refreshToken 모두 만료
  - 동작: API 요청 발생
  - 기대: localStorage에서 accessToken 제거, `/login`으로 리다이렉트

### 1.2 회원가입 (`/register`)

#### 정상 시나리오

- [ ] **SC-014**: 유효한 정보로 회원가입
  - 조건: 미사용 이메일, 유효한 비밀번호 (8자 이상, 영문+숫자+특수문자)
  - 동작: 이름, 이메일, 비밀번호, 비밀번호 확인 입력 후 "계정 만들기" 클릭
  - 기대: 성공 화면 전환 (체크 아이콘 + "회원가입이 완료되었습니다" + 입력한 이메일 표시), 인증 메일 발송 안내

- [ ] **SC-015**: 비밀번호 실시간 유효성 힌트
  - 조건: 비밀번호 필드에 입력 시작
  - 동작: 비밀번호 한 글자씩 입력
  - 기대: 4개 pill 표시 (8자 이상, 영문 포함, 숫자 포함, 특수문자 포함), 조건 충족 시 초록색 + 체크마크 전환

#### 에러 시나리오

- [ ] **SC-016**: 빈 필드 제출
  - 조건: 모든 필드가 비어있는 상태
  - 동작: "계정 만들기" 클릭
  - 기대: 각 필드별 에러 메시지 표시 (이름, 이메일, 비밀번호, 비밀번호 확인)

- [ ] **SC-017**: 잘못된 이메일 형식
  - 조건: "notanemail" 입력
  - 동작: 폼 제출
  - 기대: "올바른 이메일 형식을 입력해주세요." 에러

- [ ] **SC-018**: 비밀번호 8자 미만
  - 조건: "Ab1!" (4자) 입력
  - 동작: 폼 제출
  - 기대: "비밀번호는 8자 이상이어야 합니다." 에러

- [ ] **SC-019**: 비밀번호에 특수문자 누락
  - 조건: "Abcdefg1" (특수문자 없음)
  - 동작: 폼 제출
  - 기대: "영문, 숫자, 특수문자를 모두 포함해야 합니다." 에러

- [ ] **SC-020**: 비밀번호 확인 불일치
  - 조건: 비밀번호: "Abcdef1!", 확인: "Abcdef2!"
  - 동작: 폼 제출
  - 기대: "비밀번호가 일치하지 않습니다." 에러

- [ ] **SC-021**: 이미 사용 중인 이메일
  - 조건: 기존 가입된 이메일 주소 사용
  - 동작: 폼 제출
  - 기대: `EMAIL_EXISTS` 에러, "이미 사용 중인 이메일입니다." 서버 에러 표시

- [ ] **SC-022**: 네트워크 오류 시 회원가입
  - 조건: API 서버 연결 불가
  - 동작: 폼 제출
  - 기대: "네트워크 오류가 발생했습니다. 다시 시도해주세요." 표시

#### 엣지 케이스

- [ ] **SC-023**: 필드 수정 시 에러 메시지 자동 제거
  - 조건: 필드 에러가 표시된 상태
  - 동작: 해당 필드 값 변경
  - 기대: 해당 필드의 에러 메시지 즉시 제거

### 1.3 이메일 인증 (`/verify-email`)

#### 정상 시나리오

- [ ] **SC-024**: 유효한 토큰으로 이메일 인증
  - 조건: URL에 유효한 `?token=xxx` 포함
  - 동작: 페이지 접근
  - 기대: 로딩 스피너 → 성공 화면 (초록 체크 아이콘 + "이메일 인증이 완료되었습니다!" + 로그인 링크)

- [ ] **SC-025**: 토큰 없이 접근 (Pending 상태)
  - 조건: `?token` 파라미터 없이 `/verify-email` 접근
  - 동작: 페이지 로드
  - 기대: 메일 확인 안내 UI (메일 아이콘 + "이메일을 확인해주세요" + 재발송 버튼)

- [ ] **SC-026**: 인증 메일 재발송
  - 조건: Pending 상태에서 "인증 메일 재발송" 버튼 표시 중
  - 동작: 재발송 버튼 클릭
  - 기대: 버튼 "발송 중..." → "인증 메일을 다시 보냈습니다" (비활성화), 메일 재발송 완료

#### 에러 시나리오

- [ ] **SC-027**: 만료된 토큰
  - 조건: 만료된 인증 토큰
  - 동작: 페이지 접근
  - 기대: `TOKEN_EXPIRED` 에러, "인증 링크가 만료되었습니다. 다시 회원가입해주세요." + 회원가입 링크

- [ ] **SC-028**: 유효하지 않은 토큰
  - 조건: 변조된 토큰 값
  - 동작: 페이지 접근
  - 기대: `TOKEN_INVALID` 에러, "유효하지 않은 인증 링크입니다." 표시

- [ ] **SC-029**: 이미 인증된 이메일
  - 조건: 이미 인증 완료된 토큰으로 재접근
  - 동작: 페이지 접근
  - 기대: `ALREADY_VERIFIED` 에러, "이미 인증이 완료된 이메일입니다." + 로그인 링크

### 1.4 초대 수락 (`/invite/[token]`)

#### 정상 시나리오

- [ ] **SC-030**: 로그인 상태에서 초대 수락
  - 조건: 인증된 사용자, 유효한 초대 토큰
  - 동작: 초대 페이지 접근 → 워크스페이스 이름, 역할, 초대자 정보 확인 → "초대 수락" 클릭
  - 기대: `/{workspaceSlug}`으로 리다이렉트, 해당 워크스페이스 멤버로 추가

- [ ] **SC-031**: 초대 거절
  - 조건: 인증된 사용자, 유효한 초대
  - 동작: "거절" 버튼 클릭
  - 기대: `/workspaces`로 이동, 초대 상태 변경 없음

#### 에러 시나리오

- [ ] **SC-032**: 비로그인 상태에서 초대 접근
  - 조건: 로그인하지 않은 상태
  - 동작: `/invite/[token]` 접근
  - 기대: "초대를 수락하려면 먼저 로그인해주세요" 메시지, 로그인/회원가입 버튼 (redirect 파라미터 포함)

- [ ] **SC-033**: 만료된 초대
  - 조건: 만료된 초대 토큰
  - 동작: 페이지 접근
  - 기대: `INVITATION_EXPIRED` 에러, "이 초대 링크는 만료되었습니다." + "홈으로 이동" 버튼

- [ ] **SC-034**: 이미 멤버인 워크스페이스 초대
  - 조건: 이미 해당 워크스페이스 멤버인 사용자
  - 동작: 초대 수락 시도
  - 기대: `ALREADY_MEMBER` 에러, 2초 후 해당 워크스페이스로 자동 리다이렉트

- [ ] **SC-035**: 존재하지 않는 초대 토큰
  - 조건: 무효한 토큰 값
  - 동작: 페이지 접근
  - 기대: `INVITATION_NOT_FOUND` 에러, "유효하지 않은 초대 링크입니다." 표시

## 2. 워크스페이스 (Workspaces)

### 2.1 워크스페이스 목록 (`/workspaces`)

#### 정상 시나리오

- [ ] **SC-036**: 워크스페이스 목록 로드
  - 조건: 1개 이상의 워크스페이스 보유
  - 동작: `/workspaces` 접근
  - 기대: 각 워크스페이스별 이름, 역할 뱃지 (소유자/관리자/편집자/뷰어), 공개 여부, 마지막 활동 시간 표시

- [ ] **SC-037**: 워크스페이스에서 "문서" 버튼 클릭
  - 조건: 워크스페이스 목록 표시 중
  - 동작: 특정 워크스페이스의 "문서" 버튼 클릭
  - 기대: `/{workspaceName}/doc`으로 이동

- [ ] **SC-038**: 워크스페이스 설정 아이콘 클릭
  - 조건: 워크스페이스 목록 표시 중
  - 동작: 설정(톱니바퀴) 아이콘 클릭
  - 기대: `/{workspaceName}/settings`로 이동

#### 빈 상태

- [ ] **SC-039**: 워크스페이스가 없는 상태
  - 조건: 신규 가입 사용자, 워크스페이스 0개
  - 동작: `/workspaces` 접근
  - 기대: 빈 상태 UI ("첫 워크스페이스를 만들어보세요" + 안내 메시지), "워크스페이스 만들기" 버튼 표시 (US1-AS4)

### 2.2 워크스페이스 생성

- [ ] **SC-040**: 정상 워크스페이스 생성
  - 조건: "워크스페이스 만들기" 버튼 클릭
  - 동작: 모달에서 이름 입력 후 제출
  - 기대: 워크스페이스 생성, 목록 갱신, `/{name}/doc`으로 자동 이동

- [ ] **SC-041**: 빈 이름으로 생성 시도
  - 조건: 생성 모달 열린 상태
  - 동작: 이름 비워두고 제출
