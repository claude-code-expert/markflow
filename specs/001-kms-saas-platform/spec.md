## Clarifications

### Session 2026-03-26

- Q: 워크스페이스 공개/비공개 기본 설정은? → A: 기본 공개. 생성 시 공개 상태이며, Owner가 비공개로 전환 가능
- Q: 삭제된 문서를 사용자가 조회/복원할 수 있는 UI가 필요한가? → A: 사이드바에 휴지통 섹션 추가. 삭제 문서 목록 조회 + 복원/영구삭제 가능
- Q: 폴더 삭제 시 하위 카테고리(서브폴더)가 있으면 어떻게 처리하는가? → A: 하위 카테고리가 있으면 삭제 차단. 비어있을 때만 삭제 허용
- Q: 비밀번호 재설정 기능을 Phase 1에 포함하는가? → A: Phase 2로 연기
- Q: Phase 1 동시 접속 사용자 규모 목표는? → A: 최대 50명 동시 접속

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 회원가입 및 개인 워크스페이스 시작 (Priority: P1)

신규 사용자가 이메일로 가입하고, 자동 생성된 개인 워크스페이스에서 첫 문서를 작성한다.

**Why this priority**: 모든 기능의 전제 조건. 인증과 개인 공간 없이는 어떤 기능도 사용할 수 없다.

**Independent Test**: 회원가입 → 이메일 인증 → 로그인 → Root 워크스페이스 확인 → 첫 문서 작성 → 자동 저장 확인까지 단독 검증 가능

**Acceptance Scenarios**:

1. **Given** 미가입 사용자, **When** 이메일/비밀번호로 회원가입, **Then** 인증 이메일 발송(24시간 유효) 및 계정 생성
2. **Given** 이메일 인증 완료, **When** 로그인, **Then** Access Token(15분) + Refresh Token(7일) 발급
3. **Given** 로그인 상태, **When** 대시보드 진입, **Then** "My Notes" Root 워크스페이스가 자동 생성되어 있음
4. **Given** Root 워크스페이스, **When** 새 문서 생성 → 편집, **Then** 1초 디바운스 후 자동 저장
5. **Given** 동일 계정에 5회 연속 로그인 실패, **When** 6번째 시도, **Then** 15분 계정 잠금
6. **Given** 동일 IP에서 10회/15분 초과, **When** 추가 요청, **Then** Rate Limit 적용

### User Story 2 - 팀 워크스페이스 생성 및 멤버 관리 (Priority: P1)

사용자가 팀 워크스페이스를 만들고, 동료를 초대하여 역할을 부여하며, 외부 사용자의 가입 신청을 관리한다.

**Why this priority**: 팀 협업의 기반. 워크스페이스와 멤버 관리 없이는 KMS의 핵심 가치인 팀 지식 관리가 불가능하다.

**Independent Test**: 워크스페이스 생성 → 멤버 초대(이메일) → 초대 수락 → 역할 부여 → 권한별 접근 확인까지 단독 검증 가능

**Acceptance Scenarios**:

1. **Given** 로그인 사용자, **When** 워크스페이스 생성(이름+slug), **Then** Owner 역할로 공개 상태의 워크스페이스 생성됨
2. **Given** Owner, **When** 워크스페이스 공개/비공개 전환, **Then** 설정 변경 즉시 반영. 비공개 시 가입 신청 차단
3. **Given** Owner/Admin, **When** 이메일로 멤버 초대, **Then** 72시간 유효 초대 링크 발송
4. **Given** 초대 링크 수신자, **When** 링크 클릭 → 수락, **Then** 지정된 역할로 워크스페이스에 합류
5. **Given** 공개 워크스페이스, **When** 비멤버가 가입 신청(메시지 포함), **Then** 관리자에게 대기 상태로 노출
6. **Given** Admin/Owner, **When** 가입 신청 승인(역할 지정), **Then** 멤버 추가 + 신청자에게 알림
7. **Given** Viewer 역할, **When** 문서 생성 시도, **Then** 접근 거부

---

### User Story 3 - 문서 작성 및 카테고리 관리 (Priority: P1)

팀원이 카테고리(폴더)로 문서를 구조화하고, 문서를 작성/편집/삭제하며, 삭제된 문서를 휴지통에서 복원한다.

**Why this priority**: KMS의 핵심 기능. 문서 생성/관리와 폴더 구조 없이는 지식 관리 도구로서의 가치를 전달할 수 없다.

**Independent Test**: 폴더 생성 → 폴더 내 문서 생성 → 편집(에디터) → 자동 저장 → 버전 확인 → 삭제 → 휴지통에서 복원 → 폴더 삭제까지 단독 검증 가능

**Acceptance Scenarios**:

1. **Given** Editor 이상 역할, **When** 새 문서 생성(제목+카테고리 선택), **Then** 빈 문서 생성, slug 자동 생성, 에디터 진입
2. **Given** 문서 편집 중, **When** 입력 후 1초 경과, **Then** 자동 저장 + 새 버전 스냅샷 생성(최대 20개)
3. **Given** 사이드바, **When** 폴더 생성 버튼 클릭, **Then** 새 폴더 생성 모달(이름+상위 위치+경로 미리보기)
4. **Given** 폴더 항목, **When** 우클릭 또는 더보기 버튼, **Then** 컨텍스트 메뉴(새 문서/하위 폴더/이름 변경/삭제) 표시
5. **Given** 하위 카테고리가 없는 빈 폴더, **When** 폴더 삭제(이름 입력 확인), **Then** 폴더 제거
6. **Given** 하위 카테고리가 있는 폴더, **When** 삭제 시도, **Then** 삭제 차단 및 안내 메시지 표시
7. **Given** Editor 이상, **When** 문서 삭제, **Then** Soft Delete 처리, 사이드바 휴지통으로 이동
8. **Given** 사이드바 휴지통, **When** 삭제된 문서 목록 조회, **Then** 삭제일/원래 위치 표시, 복원/영구삭제 선택 가능
9. **Given** 휴지통 내 문서, **When** 복원 클릭, **Then** 원래 카테고리로 복원 (카테고리 삭제된 경우 루트로 복원)
10. **Given** 휴지통 내 문서, **When** 30일 경과, **Then** 영구 삭제

### User Story 4 - 문서 간 링크 및 DAG 내비게이션 (Priority: P2)

사용자가 문서 간 연관 관계(Prev/Next/연관)를 설정하고, DAG Pipeline 시각화로 문서 구조를 파악한다.

**Why this priority**: 문서 간 관계를 통해 지식 그래프 가치를 제공하지만, 기본 CRUD(US3)가 선행되어야 한다.

**Independent Test**: 문서 3개 생성 → Prev/Next 설정 → 연관 문서 추가 → 메타 패널 미니 DAG 확인 → 프리뷰 하단 DAG 내비 확인 → 그래프 뷰 페이지 확인까지 단독 검증 가능

**Acceptance Scenarios**:

1. **Given** 문서 A, **When** Prev/Next 문서 설정, **Then** 양방향 관계 저장, 순환 참조 방지
2. **Given** 문서 A, **When** 연관 문서 추가(최대 20개), **Then** 연관 관계 저장, 초과 시 에러
3. **Given** 문서 편집 화면, **When** 메타 패널 확인, **Then** 미니 DAG 그래프에 문서 관계 표시
4. **Given** 프리뷰 하단, **When** DAG 내비게이션 확인, **Then** 이전/현재/다음/연관 문서 파이프라인 표시
5. **Given** 사이드바, **When** 그래프 뷰 클릭, **Then** 워크스페이스 전체 문서 관계를 DAG Pipeline으로 표시
6. **Given** DAG 그래프 내 노드, **When** 클릭, **Then** 해당 문서 에디터로 이동

---

### User Story 5 - Import/Export (Priority: P2)

사용자가 마크다운 파일을 가져오거나, 작성한 문서를 다양한 형식으로 내보낸다.

**Why this priority**: 데이터 이동성은 사용자 신뢰 확보에 필수이지만, 문서 관리(US3)가 선행되어야 한다.

**Independent Test**: .md 파일 Import → 문서 확인 → .md Export → 내용 일치 검증까지 단독 검증 가능

**Acceptance Scenarios**:

1. **Given** 워크스페이스, **When** .md 파일 업로드, **Then** 문서로 변환되어 저장
2. **Given** 워크스페이스, **When** .zip 파일 업로드(폴더 구조 포함), **Then** 카테고리 구조 유지하며 문서 생성
3. **Given** 문서, **When** .md Export, **Then** 마크다운 파일 다운로드
4. **Given** 카테고리, **When** .zip Export, **Then** 폴더 구조 보존한 ZIP 다운로드

---

### User Story 6 - 역할 기반 권한 제어 (Priority: P1)

워크스페이스 내 모든 작업이 역할(Owner/Admin/Editor/Viewer)에 따라 통제된다.

**Why this priority**: 보안의 기본 요소이며 팀 협업 기능(US2, US3)과 동시에 구축되어야 한다.

**Independent Test**: 4가지 역할 각각으로 문서 CRUD, 멤버 관리, 설정 변경 시도 → 허용/거부 확인까지 단독 검증 가능

**Acceptance Scenarios**:

1. **Given** Owner, **When** 모든 작업 시도, **Then** 전체 허용 (삭제 포함)
2. **Given** Admin, **When** 멤버 초대/가입 신청 승인, **Then** 허용. 워크스페이스 삭제 시 거부
3. **Given** Editor, **When** 문서 읽기/쓰기, **Then** 허용. 멤버 초대/설정 변경 시 거부
4. **Given** Viewer, **When** 문서 읽기, **Then** 허용. 문서 생성/수정/삭제 시 거부
5. **Given** 모든 요청, **When** workspace_id 범위 밖 데이터 접근 시도, **Then** 거부 (데이터 격리)

---

### User Story 7 - 태그 관리 (Priority: P2)

사용자가 문서에 태그를 추가/삭제하여 분류하고 필터링한다.

**Why this priority**: 문서 관리(US3)의 확장 기능으로, 핵심 CRUD 이후 추가한다.

**Independent Test**: 문서에 태그 추가 → 태그로 필터링 → 태그 삭제까지 단독 검증 가능

**Acceptance Scenarios**:

1. **Given** 문서, **When** 태그 추가/교체, **Then** 태그 전체 교체 저장
2. **Given** 문서, **When** 31개 이상 태그 설정 시도, **Then** 에러 반환
3. **Given** 워크스페이스 문서 목록, **When** 태그 필터 적용, **Then** 해당 태그의 문서만 표시

---

### Edge Cases

- 동시에 같은 문서를 두 사용자가 편집하면 어떻게 되는가? → Phase 1에서는 마지막 저장이 우선, Phase 2에서 Yjs CRDT로 해결
- 워크스페이스 Owner가 탈퇴하면? → Owner 이전 필수, 미이전 시 탈퇴 차단
- Root 워크스페이스 삭제 시도? → 삭제 불가, 에러 반환
- 초대 링크 만료(72시간) 후 클릭? → 410 Gone 반환, 재초대 안내
- 이미 멤버인 사용자가 초대 수락? → 409 충돌 반환
- 카테고리를 자기 자신의 자손으로 이동 시도? → Phase 1에서는 카테고리 이동 미지원. Phase 2에서 이동 기능 추가 시 순환 구조 방지 필요
- 연관 문서 21개 추가 시도? → 상한 초과 에러
- 이메일 미인증 계정으로 로그인 시도? → 인증 필요 안내
- 비공개 워크스페이스에 가입 신청 시도? → 가입 신청 버튼 미노출, 직접 접근 시 거부
- 삭제된 문서의 원래 카테고리가 이미 삭제된 경우 복원? → 루트 카테고리로 복원
- 하위 카테고리가 있는 폴더 삭제 시도? → 삭제 차단, "하위 폴더를 먼저 삭제하세요" 안내

## Requirements *(mandatory)*

### Functional Requirements

**인증 & 계정**

- **FR-001**: 시스템은 이메일/비밀번호 회원가입을 지원해야 한다 (비밀번호: 8자 이상, 영문+숫자+특수문자)
- **FR-002**: 시스템은 이메일 인증 링크를 발송해야 한다 (24시간 유효)
- **FR-003**: 시스템은 토큰 기반 인증을 적용해야 한다 (단기 Access Token + 장기 Refresh Token)
- **FR-004**: 시스템은 동일 계정 5회 연속 로그인 실패 시 15분 계정 잠금을 적용해야 한다
- **FR-005**: 시스템은 인증 엔드포인트(/auth/*)에 IP 기준 10회/15분 초과 시 요청 제한을 적용해야 한다
- **FR-006**: 사용자는 프로필(이름, 아바타)을 편집할 수 있어야 한다 (아바타: JPG/PNG, 2MB 이하)
- **FR-007**: 시스템은 로그아웃 시 Refresh Token을 즉시 무효화해야 한다
- **FR-008**: 이메일 미인증 계정은 로그인이 차단되어야 한다

**워크스페이스 & 멤버**

- **FR-009**: 회원가입 완료 시 개인 Root 워크스페이스가 자동 생성되어야 한다
- **FR-010**: Root 워크스페이스는 삭제 불가이며, 이름 변경만 가능해야 한다
- **FR-011**: 사용자는 팀 워크스페이스를 생성할 수 있어야 한다 (고유 slug 기반 URL, 기본 공개 상태)
- **FR-012**: Owner는 워크스페이스의 공개/비공개를 전환할 수 있어야 한다. 비공개 시 가입 신청이 차단된다
- **FR-013**: Owner/Admin은 이메일로 멤버를 초대할 수 있어야 한다 (72시간 유효)
- **FR-014**: 비멤버는 공개 워크스페이스에 가입 신청(메시지 포함)할 수 있어야 한다
- **FR-015**: Admin/Owner는 가입 신청을 역할 지정 후 승인/거절할 수 있어야 한다 (일괄 처리 지원)
- **FR-016**: 시스템은 4가지 역할(Owner/Admin/Editor/Viewer)에 따른 권한 제어를 적용해야 한다
- **FR-017**: 모든 데이터 쿼리에 workspace_id 범위를 강제하여 데이터 격리를 보장해야 한다
- **FR-018**: 워크스페이스 리스트는 행 기반 뷰로 역할 배지와 활동 시간을 표시해야 한다
- **FR-019**: 워크스페이스 삭제 시 이름 입력 확인이 필요하며, Root 워크스페이스 삭제는 불가해야 한다
- **FR-019a**: Owner는 워크스페이스의 다른 Admin에게 Owner 역할을 이전할 수 있어야 한다. 미이전 상태에서 Owner 탈퇴는 차단된다

**문서 관리**

- **FR-020**: Editor 이상 역할은 문서를 생성할 수 있어야 한다 (제목, slug 자동 생성, 카테고리 선택)
- **FR-021**: 문서 편집 시 1초 디바운스 후 자동 저장되어야 한다
- **FR-022**: 저장 시 버전 스냅샷이 생성되어야 한다 (Phase 1: 최대 20개 보관)
- **FR-023**: 문서 삭제는 Soft Delete이며, 30일 후 영구 삭제되어야 한다
- **FR-024**: 사이드바에 휴지통 섹션을 제공해야 한다. 삭제된 문서 목록 조회, 복원, 영구삭제를 지원한다
- **FR-025**: 휴지통에서 복원 시 원래 카테고리로 복원한다. 원래 카테고리가 삭제된 경우 루트로 복원한다
- **FR-026**: 사용자는 문서에 태그를 추가/삭제할 수 있어야 한다 (최대 30개)
- **FR-027**: 문서 목록은 리스트/그리드 뷰, 정렬(이름/수정일/생성일), 필터를 지원해야 한다

**카테고리(폴더)**

- **FR-028**: 사용자는 폴더를 생성/이름 변경/삭제할 수 있어야 한다
- **FR-029**: 폴더 삭제는 하위 카테고리가 없는 경우에만 허용한다. 하위 카테고리가 있으면 삭제를 차단하고 안내 메시지를 표시한다
- **FR-030**: 폴더 삭제 시 하위 문서가 있으면 루트로 이동한 후 폴더를 제거한다. 하위 문서가 없으면 즉시 제거한다
- **FR-031**: 폴더는 무제한 중첩을 지원해야 한다
- **FR-032**: 사이드바에서 폴더 항목에 컨텍스트 메뉴(새 문서/하위 폴더/이름 변경/삭제)를 제공해야 한다
- **FR-033**: 새 폴더 생성 시 경로 미리보기를 실시간으로 표시해야 한다
- **FR-034**: 폴더 삭제 시 이름 입력 확인(안전장치)을 제공해야 한다

**문서 간 링크**

- **FR-035**: 문서 간 Prev/Next 관계를 설정할 수 있어야 한다 (순환 참조 방지)
- **FR-036**: 문서 간 연관 관계를 설정할 수 있어야 한다 (최대 20개)
- **FR-037**: 문서 관계를 파이프라인 형태로 시각화해야 한다 (메타 패널 미니 뷰 + 프리뷰 하단 내비게이션)
- **FR-038**: 워크스페이스 전체 문서 관계를 그래프 뷰 페이지로 제공해야 한다

**Import/Export**

- **FR-039**: .md 파일 단일 Import를 지원해야 한다
- **FR-040**: .zip 파일 Import(폴더 구조 유지)를 지원해야 한다
- **FR-041**: .md 파일 단일 Export를 지원해야 한다
- **FR-042**: 카테고리 전체 .zip Export(폴더 구조 보존)를 지원해야 한다

**보안**

- **FR-043**: CSRF 방어를 위한 쿠키 정책과 Origin 검증을 적용해야 한다
- **FR-044**: 모든 시크릿은 환경 변수로만 관리하고 코드에 하드코딩하지 않아야 한다

### Key Entities

- **User**: 계정 정보, 인증 자격증명, 프로필 (이름, 아바타). 여러 워크스페이스에 소속 가능
- **Workspace**: 팀/개인 지식 공간. slug 기반 URL, 멤버 목록, 설정, 공개/비공개 상태(기본 공개) 포함
- **WorkspaceMember**: User와 Workspace의 관계. 역할(Owner/Admin/Editor/Viewer) 포함
- **Document**: 마크다운 문서. 제목, slug, 내용, 카테고리 귀속, 버전 이력, 태그, 삭제 상태(is_deleted, deleted_at) 포함
- **Category**: 폴더 구조. 이름, 부모-자식 관계, 워크스페이스 귀속
- **DocumentRelation**: 문서 간 관계 (prev/next/related). 순환 참조 방지
- **Invitation**: 멤버 초대. 이메일, 토큰, 만료일, 상태
- **JoinRequest**: 가입 신청. 신청자, 메시지, 상태(대기/승인/거절)
- **DocumentVersion**: 문서 버전 스냅샷. 내용, 버전 번호, 생성 시각
- **Tag**: 문서 분류용 태그. 문서당 최대 30개

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자가 가입부터 첫 문서 작성까지 3분 이내에 완료할 수 있다
- **SC-002**: 팀 관리자가 멤버 초대부터 상대방 합류까지 5분 이내에 완료할 수 있다
- **SC-003**: 문서 편집 내용이 자동 저장까지 2초 이내에 반영된다
- **SC-004**: 사이드바 폴더 트리에서 3단계 이상 중첩된 카테고리를 탐색할 수 있다
- **SC-005**: 워크스페이스당 최소 1,000개 문서를 저장하고 목록 조회가 가능하다
- **SC-006**: 4가지 역할의 권한 매트릭스가 모든 작업에서 정확히 적용된다
- **SC-007**: 문서 간 Prev/Next/연관 관계 설정 시 순환 참조가 100% 방지된다
- **SC-008**: .md Import 후 원본과 동일한 마크다운 내용이 유지된다
- **SC-009**: 계정 잠금 및 요청 제한이 설정된 임계값에서 정확히 작동한다
- **SC-010**: 팀 내부 5명 이상이 2주간 일상 문서 작업에 사용할 수 있다
- **SC-011**: 최대 50명 동시 접속 시 API p95 응답 시간 500ms 이하, 에러율 1% 미만을 유지한다

## Assumptions

- 사용자는 안정적인 인터넷 연결 환경에서 사용한다
- Phase 1은 팀 내부 검증 목적이므로 동시 편집 충돌은 마지막 저장 우선으로 처리한다 (Phase 2에서 CRDT 도입)
- 이메일 발송은 Phase 1에서 콘솔 로그로 대체 가능하다 (Phase 2에서 실제 발송)
- 에러 모니터링은 Phase 1에서 기본 로깅, Phase 2에서 전문 모니터링 도구 연동
- 검색은 Phase 1에서 제목 부분 일치 검색, Phase 2에서 전문 검색으로 확장
- 소셜 로그인(Google/GitHub)은 Phase 2 범위이며 Phase 1에서는 이메일/비밀번호만 지원
- 비밀번호 재설정(셀프서비스)은 Phase 2 범위이며 Phase 1에서는 관리자 수동 초기화로 대응한다
- 모바일 반응형은 Phase 3 범위이며, Phase 1은 데스크탑 브라우저 기준
- `@markflow/editor` 에디터 패키지는 이미 구현 완료(Phase 0)되어 있으며 KMS에서 재사용한다
- Remember me 선택 시 Refresh Token 유효기간을 30일로 연장한다
