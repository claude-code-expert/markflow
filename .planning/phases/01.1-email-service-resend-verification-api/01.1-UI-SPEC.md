# Phase 01.1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Assessment

**This phase is predominantly backend.** The UI surface is limited to:

1. **Frontend fix:** `verify-email/page.tsx` `handleResend()` must pass `{ email }` in POST body
2. **Email HTML templates:** 3 inline-CSS HTML email templates (verification, password reset, invitation)
3. **No new pages or components** are created

All design tokens, layout, typography, and interaction patterns are already established in the existing `verify-email/page.tsx` and `globals.css`. This contract codifies those existing values as the source of truth for the email templates and the frontend fix.

## Frontend Fix Contract

### Change: `handleResend()` in verify-email/page.tsx

**Current behavior:** POST `/auth/resend-verification` with no body. The page has no email state.

**Required behavior:** POST `/auth/resend-verification` with `{ email: string }` body.

**Email source strategy (from RESEARCH.md Open Question #1):**

The register success screen in `register/page.tsx` already displays the user's email (`{email}` state). When the user navigates from register success to verify-email, the email must be transferred. Options ranked:

1. **(Recommended) Register success page links to `/verify-email?email=xxx`** -- simplest, email is visible in URL but this is the user's own address during their own registration flow. Add `email` to searchParams in verify-email page.
2. localStorage: works across tabs but adds storage dependency.

The verify-email page must:
- Read `email` from `searchParams` (alongside existing `token` param)
- Pass `email` to `handleResend()` body: `body: { email }`
- If `email` param is missing in pending state, show the resend button disabled with helper text: `이메일 주소를 확인할 수 없습니다. 로그인 페이지에서 다시 시도해주세요.`

### No Visual Changes

The verify-email page layout, colors, typography, spacing, icons, and interaction states remain exactly as-is. Only the data flow changes (email passed through URL param to API body).

## Interaction States

All states already exist in `verify-email/page.tsx`. No new states are added.

| State | Trigger | Visual |
|-------|---------|--------|
| `pending` | No `token` in URL | Mail icon (teal accent), heading, info box, resend button |
| `loading` | `token` present, verifying | Spinner icon (blue accent), "인증 중..." |
| `success` | Verification API 200 | Green check icon, success message, login CTA |
| `error` | Verification API error | Red X icon, error message, contextual CTA |
| Resend loading | Button click | Button text: "발송 중...", opacity 0.6, cursor not-allowed |
| Resend success | API 200 | Button text: "인증 메일을 다시 보냈습니다", disabled |
