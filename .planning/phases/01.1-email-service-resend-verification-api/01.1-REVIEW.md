# Phase 01.1: Code Review Report

**Reviewed:** 2026-04-13T12:00:00Z
**Depth:** standard
**Files Reviewed:** 9
**Status:** issues_found

## Summary

Reviewed the email service, resend-verification API, auth routes, and frontend auth pages. The backend implementation is generally solid: email enumeration prevention is correctly implemented in `resendVerification`, token rotation works properly, and the invitation email template correctly escapes user-supplied strings. However, there is a critical HTTP method mismatch between the frontend verify-email page and the backend API route that will cause verification to fail in production. Additionally, URL parameters in email templates lack HTML escaping, and there are a few convention violations.

## Critical Issues

### CR-01: HTTP Method Mismatch -- verify-email frontend sends POST to a GET endpoint

**File:** `apps/web/app/(auth)/verify-email/page.tsx:200-203`
**Issue:** The frontend calls `apiFetch('/auth/verify-email?token=...', { method: 'POST' })` but the backend route at `apps/api/src/routes/auth.ts:54` registers this as `app.get('/verify-email', ...)`. This mismatch means email verification will always fail with a 404 or 405 error -- users cannot complete registration.
**Fix:**
```typescript
// verify-email/page.tsx line 200-203 -- change POST to GET
await apiFetch<VerifyEmailResponse>(
  `/auth/verify-email?token=${encodeURIComponent(token as string)}`,
  { method: 'GET' },
);
```

## Warnings

### WR-01: URL parameters not HTML-escaped in email templates

**File:** `apps/api/src/utils/email.ts:52`
**Issue:** `verificationEmailHtml(verifyUrl)` interpolates `verifyUrl` directly into an `href` attribute without escaping. While the URL is currently constructed from `FRONTEND_URL` (an env var) and a UUID token, if `FRONTEND_URL` ever contains `"` or other HTML-special characters, it could break the template or enable HTML injection. The same applies to `passwordResetEmailHtml` at line 61. Note that `invitationEmailHtml` correctly escapes its user-supplied parameters via `escapeHtml()`.
**Fix:**
```typescript
export function verificationEmailHtml(verifyUrl: string): string {
  const safeUrl = escapeHtml(verifyUrl);
  return `...
  <a href="${safeUrl}" ...>이메일 인증하기</a>
  ...`;
}
```
Apply the same pattern to `passwordResetEmailHtml`.

### WR-02: Silent error swallowing in resend handler

**File:** `apps/web/app/(auth)/verify-email/page.tsx:239`
**Issue:** The `handleResend` function has an empty `catch` block that silently discards errors. If the resend API call fails (network error, server error), the user gets no feedback -- the button just re-enables with no indication of failure.
**Fix:**
```typescript
} catch {
  // Show user-facing error feedback
  setMessage('인증 메일 발송에 실패했습니다. 다시 시도해주세요.');
}
```

### WR-03: Unsafe type assertion bypasses request body validation

**File:** `apps/api/src/routes/auth.ts:87`
**Issue:** The login route uses `request.body as { email: string; password: string; rememberMe?: boolean }` instead of using Fastify's generic type parameter pattern (as used correctly in `/register`, `/resend-verification`, etc.). This `as` assertion silently bypasses TypeScript's type checking -- if `request.body` has a different shape at runtime, the destructured values will be `undefined` without compile-time warning.
**Fix:**
```typescript
app.post<{ Body: { email: string; password: string; rememberMe?: boolean } }>(
  '/login',
  authRateLimit(),
  async (request, reply) => {
    const { email, password, rememberMe } = request.body;
    // ...
  },
);
```

## Info

### IN-01: Default export used in page components (convention violation)

**File:** `apps/web/app/(auth)/register/page.tsx:291`
**File:** `apps/web/app/(auth)/verify-email/page.tsx:171`
**Issue:** Project conventions state "default export 사용 금지", but both auth page components use `export default function`. Note: Next.js App Router requires default exports for `page.tsx` files, so this may be an intentional exception to the convention. If so, the convention should be updated to clarify the exception for Next.js page/layout files.
**Fix:** Either update `CLAUDE.md` to note the Next.js page exception, or restructure to use named exports with a default re-export wrapper if technically feasible.

### IN-02: TOKEN_EXPIRED error suggests re-registration instead of resend

**File:** `apps/web/app/(auth)/verify-email/page.tsx:16`
**Issue:** When `TOKEN_EXPIRED` is returned, the error message says "다시 회원가입해주세요" (please register again) and the CTA links to `/register` (line 345). Since the resend-verification endpoint exists and issues a new token, the user should be directed to resend verification instead of re-registering.
**Fix:**
```typescript
case 'TOKEN_EXPIRED':
  return '인증 링크가 만료되었습니다. 인증 메일을 다시 받아주세요.';
```
And update the CTA button at line 345 to trigger resend or link to a resend flow.
