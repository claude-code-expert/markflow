# Phase 01.1: Email Service & Resend Verification API - Research

**Researched:** 2026-04-13
**Domain:** Email service integration (Resend SDK) + verification resend API
**Confidence:** HIGH

## Summary

Phase 01.1은 현재 `logger.info()`로만 출력되는 3가지 이메일(가입 인증, 비밀번호 재설정, 초대)을 Resend 서비스로 교체하고, 이메일 인증 재발송 API를 추가하는 작업이다. Resend SDK v4.x (최신 npm 4.1.2)는 TypeScript 네이티브이며 `resend.emails.send()` 단일 메서드로 HTML 이메일 발송을 지원한다.

기존 코드 분석 결과, auth-service.ts의 `register()` (85행)와 `forgotPassword()` (263행), invitation-service.ts의 `create()` (45행) 총 3곳의 `logger.info()` 호출을 Resend SDK 호출로 교체해야 한다. 재발송 API는 프론트엔드 verify-email 페이지에서 이미 `POST /auth/resend-verification` 호출 코드가 존재하므로, 백엔드 엔드포인트만 추가하면 된다.

**중요 발견:** 프론트엔드 verify-email 페이지의 `handleResend()` 함수가 body 없이 POST 요청을 보내고 있다. 회원가입 직후 사용자는 로그인하지 않은 상태(register는 토큰을 반환하지 않음)이므로, Authorization 헤더도 없다. 재발송 API는 `{ email }` body를 받아야 하며, 프론트엔드도 email을 전달하도록 수정이 필요하다.

**Primary recommendation:** Resend SDK를 auth-service.ts에 직접 통합하고, `RESEND_API_KEY` 미설정 시 logger.info() 폴백 패턴을 사용하라. 재발송 API에는 email body 파라미터를 요구하고 프론트엔드를 함께 수정하라.

### Pattern 1: Email Helper with Fallback (Discretion Area)
**What:** Resend SDK 싱글턴 + sendEmail() 헬퍼 함수
**When to use:** 모든 이메일 발송 시
**Recommendation:** `utils/email.ts`에 싱글턴 Resend 인스턴스와 sendEmail 헬퍼를 배치. D-02에서 "추상화 레이어 없이 직접 사용"이라고 했으나, 3곳에서 반복되는 Resend 초기화 + 폴백 로직은 헬퍼로 추출하는 것이 합리적이다 (추상화가 아닌 DRY).

```typescript
// Source: Resend official docs (https://resend.com/docs/send-with-nodejs)
// + D-07 폴백 패턴 적용
import { Resend } from 'resend';
import { logger } from './logger.js';

const RESEND_API_KEY = process.env.RESEND_API_KEY;
const EMAIL_FROM = process.env.EMAIL_FROM ?? 'MarkFlow <noreply@markflow.app>';

// 싱글턴: API 키 없으면 null (폴백 모드)
const resend = RESEND_API_KEY ? new Resend(RESEND_API_KEY) : null;

interface SendEmailParams {
  to: string;
  subject: string;
  html: string;
}

export async function sendEmail(params: SendEmailParams): Promise<void> {
  if (!resend) {
    // D-07: 폴백 모드 -- 개발 환경에서 콘솔 출력
    logger.info(`[EMAIL FALLBACK] To: ${params.to}, Subject: ${params.subject}`);
    logger.info(`[EMAIL FALLBACK] HTML: ${params.html}`);
    return;
  }

  const { data, error } = await resend.emails.send({
    from: EMAIL_FROM,
    to: [params.to],
    subject: params.subject,
    html: params.html,
  });

  if (error) {
    logger.error(`Failed to send email to ${params.to}`, { error });
    throw new Error(`Email send failed: ${error.message}`);
  }

  logger.info(`Email sent to ${params.to}`, { id: data?.id });
}
```

### Pattern 4: Resend Verification Service Logic (Discretion Area)
**Recommendation:** 새 토큰을 발급하라.

재발송 시 기존 토큰 재사용 vs 새 토큰 발급:
- **새 토큰 발급 추천 이유:** 기존 토큰이 만료되었을 수 있음 (24시간). 새 토큰을 발급하면 항상 유효한 링크를 보장.
- 기존 토큰이 아직 유효해도 새 토큰으로 교체하는 것이 보안상 안전 (이전 링크 무효화).

```typescript
async function resendVerification(email: string) {
  const found = await db
    .select()
    .from(users)
    .where(eq(users.email, email.toLowerCase()))
    .limit(1);

  const user = found[0];

  // 사용자가 없거나 이미 인증된 경우에도 동일 응답 (이메일 열거 방지)
  if (!user || user.emailVerified) {
    return { sent: true };
  }

  // 새 토큰 발급
  const emailVerifyToken = crypto.randomUUID();
  const emailVerifyExpiresAt = new Date(Date.now() + 24 * 60 * 60 * 1000);

  await db
    .update(users)
    .set({
      emailVerifyToken,
      emailVerifyExpiresAt,
      updatedAt: new Date(),
    })
    .where(eq(users.id, user.id));

  // 이메일 발송
  const verifyUrl = `${process.env.FRONTEND_URL}/verify-email?token=${emailVerifyToken}`;
  await sendEmail({
    to: user.email,
    subject: 'MarkFlow 이메일 인증',
    html: verificationEmailHtml(verifyUrl),
  });

  return { sent: true };
}
```

### Anti-Patterns to Avoid
- **API 키를 코드에 하드코딩:** 절대 금지. 환경변수만 사용.
- **이메일 열거 공격 허용:** 사용자 존재 여부를 응답에서 구분할 수 있으면 안 됨. 항상 `{ sent: true }` 반환.
- **폴백 없는 Resend 의존:** API 키 미설정 시 서버 크래시 방지. null 체크 필수.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SMTP 연결 관리 | 자체 SMTP 클라이언트 | Resend SDK | 연결 풀링, 재시도, 에러 핸들링 내장 |
| 이메일 템플릿 엔진 | 자체 템플릿 시스템 | 순수 HTML 문자열 | D-04 결정. 3개 템플릿에 엔진 불필요 |
| Rate limiting | 자체 구현 | @fastify/rate-limit | 이미 설치/사용 중 |

## Common Pitfalls

### Pitfall 1: 프론트엔드-백엔드 API 불일치
**What goes wrong:** 프론트엔드 verify-email 페이지의 `handleResend()`가 body 없이 POST를 보내는데, 백엔드가 `{ email }` body를 요구하면 400 에러 발생.
**Why it happens:** 프론트엔드가 이미 구현되어 있고, 백엔드 API가 아직 없는 상태에서 API 계약이 불일치.
**How to avoid:** 프론트엔드 `handleResend()`를 수정하여 email을 body에 포함하거나, 또는 verify-email 페이지에 email state를 전달하는 방법 필요. 현재 verify-email 페이지는 email 정보를 갖고 있지 않으므로, register 성공 후 redirect 시 email을 query param이나 state로 전달해야 한다.
**Warning signs:** 재발송 버튼 클릭 시 콘솔에 400/422 에러.

### Pitfall 2: 환경변수 누락으로 프로덕션에서 이메일 미발송
**What goes wrong:** 배포 시 RESEND_API_KEY를 설정하지 않으면 폴백 모드로 동작하여 실제 이메일이 발송되지 않음.
**Why it happens:** D-07 폴백 패턴이 개발 편의를 위한 것이지만, 프로덕션에서도 조용히 실패.
**How to avoid:** 프로덕션 환경(NODE_ENV=production)에서 RESEND_API_KEY가 없으면 서버 시작 시 경고 로그 출력. 또는 환경변수 검증 로직 추가.
**Warning signs:** 사용자가 인증 이메일을 받지 못한다는 리포트.

### Pitfall 3: 이메일 열거 공격 (Email Enumeration)
**What goes wrong:** 재발송 API 응답이 사용자 존재 여부를 노출하면, 공격자가 등록된 이메일 목록을 수집 가능.
**Why it happens:** "사용자를 찾을 수 없습니다" vs "이메일을 보냈습니다" 같은 다른 응답을 반환할 때.
**How to avoid:** 항상 동일한 성공 응답 `{ sent: true }` 반환. forgotPassword()에서 이미 이 패턴을 사용 중 (auth-service.ts:248).
**Warning signs:** 보안 감사에서 지적.

### Pitfall 4: Resend SDK 에러 핸들링
**What goes wrong:** Resend SDK는 try/catch 대신 `{ data, error }` 패턴을 사용. try/catch로 감싸면 에러를 놓칠 수 있음.
**Why it happens:** 일반적인 Node.js 라이브러리와 다른 에러 패턴.
**How to avoid:** 항상 `error` 필드를 체크. SDK 공식 문서에서도 try/catch 사용을 권장하지 않음.
**Warning signs:** 이메일 발송 실패가 로그에 안 나타남.

### Pitfall 5: FRONTEND_URL 환경변수 누락
**What goes wrong:** 이메일에 포함되는 인증/재설정 링크의 base URL이 undefined가 되어 깨진 링크 발송.
**Why it happens:** 기존에는 logger.info()로 상대 경로만 출력했으나, 실제 이메일에는 전체 URL이 필요.
**How to avoid:** FRONTEND_URL 환경변수 추가 및 .env.example에 문서화. 미설정 시 기본값 `http://localhost:3000` 사용.
**Warning signs:** 이메일 내 링크가 작동하지 않음.

async function handleResend() {

  setResendSuccess(false);

    await apiFetch<VerifyEmailResponse>('/auth/resend-verification', {
      method: 'POST',
      body: { email: userEmail }, // email 파라미터 필요

    setResendSuccess(false);
  } finally {
    setIsResending(false);
  }
}
