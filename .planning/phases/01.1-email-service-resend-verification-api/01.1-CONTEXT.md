# Phase 01.1: Email Service & Resend Verification API - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

현재 `logger.info()`로만 출력되는 3가지 이메일(가입 인증, 비밀번호 재설정, 초대)을 Resend 서비스로 교체하고, 이메일 인증 재발송 API(POST /auth/resend-verification)를 추가한다. EMAIL-01, EMAIL-02 요구사항을 구현한다.

</domain>

<decisions>
## Implementation Decisions

### 이메일 서비스 선택
- **D-01:** **Resend** 사용 — TypeScript SDK 네이티브, 무료 3,000통/월, Next.js/Vercel 생태계와 호환. `resend` npm 패키지 설치.
- **D-02:** auth-service.ts에서 **Resend SDK 직접 호출** — EmailService 추상화 레이어 없이 직접 사용. MVP 단계에서 불필요한 추상화 방지. 필요 시 나중에 추출 가능.

### 이메일 교체 범위
- **D-03:** `logger.info()` 3곳 **모두 교체** — 가입 인증(auth-service.ts:85), 비밀번호 재설정(auth-service.ts:263), 초대(invitation-service.ts:45) 모두 Resend로 실제 이메일 발송.
- **D-04:** 이메일 템플릿은 **기본 HTML** (inline CSS) — 로고/브랜드 이미지 없이 텍스트 + 버튼 + 링크 수준. React Email 미사용.

### 재발송 API
- **D-05:** `POST /api/v1/auth/resend-verification` 엔드포인트 추가 — 이메일 미인증 사용자가 인증 이메일을 다시 요청할 수 있다.
- **D-06:** 재발송 rate limit은 기존 `authRateLimit()` 패턴 적용 (IP 기준 10회/15분).

### 개발 환경 전략
- **D-07:** `RESEND_API_KEY` 환경변수 미설정 시 **logger.info() 폴백** 유지 — 로컬 개발 환경에서 API 키 없이도 기존처럼 콘솔 출력. 설정 시 Resend로 실제 발송.
- **D-08:** `.env.example`에 `RESEND_API_KEY`, `EMAIL_FROM` 환경변수 문서화.
- **D-09:** 통합 테스트에서는 항상 폴백 모드 사용 (환경변수 미설정).

### Claude's Discretion
- Resend SDK 초기화 위치 및 싱글턴 패턴
- HTML 이메일 템플릿 구체적 마크업 구조
- 재발송 시 기존 토큰 재사용 vs 새 토큰 발급 결정
- 이미 인증된 사용자의 재발송 요청 시 에러 응답 코드

</decisions>

## Existing Code Insights

### Reusable Assets
- `authRateLimit()` — auth.ts에 정의된 rate limit 헬퍼. 재발송 API에 동일 적용.
- `badRequest()`, `unauthorized()`, `gone()` — 에러 유틸리티. 만료 토큰, 이미 인증된 사용자 등 에러 처리에 재사용.

- **Fastify 라우트**: `app.post('/path', authRateLimit(), async (request, reply) => { ... })` 패턴
