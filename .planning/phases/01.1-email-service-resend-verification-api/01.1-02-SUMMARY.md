# Phase 01.1 Plan 02: Resend Verification API + Frontend Wiring Summary

POST /resend-verification endpoint with email enumeration prevention, TDD integration tests, and frontend verify-email page wired to pass email param to API.

## What Was Built

### Task 1: resendVerification() Service + POST /resend-verification Route + Integration Tests (TDD)

**Service method** (`auth-service.ts`):
- `resendVerification(email)` looks up user by email, returns `{ sent: true }` for all cases (non-existent, already verified, or unverified)
- For unverified users: issues new `emailVerifyToken` (UUID v4), sets 24h expiry, sends verification email via `sendEmail()`
- Previous token is invalidated on resend (security: old links become invalid)

**Route** (`auth.ts`):
- `POST /resend-verification` with `authRateLimit()` (10 req/15min per IP)
- Validates `email` body field, returns 400 `MISSING_FIELDS` if absent

**Integration tests** (5 scenarios, all passing):
1. Unverified user resend: new token issued, different from original
2. Non-existent email: 200 `{ sent: true }` (enumeration prevention)
3. Already verified user: 200 `{ sent: true }`, token unchanged (null)
4. Missing email body: 400 `MISSING_FIELDS`
5. Verify with new token after resend: successful verification

### Task 2: Frontend verify-email + register Page Modifications

**verify-email/page.tsx**:
- Extracts `email` from searchParams alongside `token`
- `handleResend()` sends `{ email: userEmail }` in POST body
- Resend button disabled when `userEmail` is undefined
- Shows guidance text when email is unavailable: "이메일 주소를 확인할 수 없습니다"

**register/page.tsx**:
- Success screen adds "이메일 확인 페이지로 이동" link: `/verify-email?email=${encodeURIComponent(email)}`
- Placed above existing "로그인 페이지로 이동" link
