### CR-01: HTTP Method Mismatch -- verify-email frontend sends POST to a GET endpoint

**Files modified:** `apps/web/app/(auth)/verify-email/page.tsx`
**Commit:** c1c8f56
**Applied fix:** Changed `{ method: 'POST' }` to `{ method: 'GET' }` in the `verify()` function's `apiFetch` call (line 202). The backend registers `/verify-email` as a GET route, so the frontend must match.

### WR-01: URL parameters not HTML-escaped in email templates

**Files modified:** `apps/api/src/utils/email.ts`
**Commit:** 507a2de
**Applied fix:** Added `const safeUrl = escapeHtml(verifyUrl)` in `verificationEmailHtml()` and `const safeUrl = escapeHtml(resetUrl)` in `passwordResetEmailHtml()`, then used `safeUrl` in the `href` attribute interpolation. This matches the pattern already used in `invitationEmailHtml()`.

### WR-02: Silent error swallowing in resend handler

**Files modified:** `apps/web/app/(auth)/verify-email/page.tsx`
**Commit:** 068501e
**Applied fix:** Added `setMessage('...')` call in the empty `catch` block of `handleResend()` to provide user-facing error feedback when the resend API call fails.

### WR-03: Unsafe type assertion bypasses request body validation

**Files modified:** `apps/api/src/routes/auth.ts`
**Commit:** 404718f
**Applied fix:** Replaced `app.post('/login', ...)` with `app.post<{ Body: { email: string; password: string; rememberMe?: boolean } }>('/login', ...)` and removed the `as` type assertion and explicit `FastifyRequest`/`FastifyReply` annotations on the handler parameters. This matches the pattern used in all other routes in the file.
