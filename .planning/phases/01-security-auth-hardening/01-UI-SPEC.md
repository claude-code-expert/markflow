## Component Inventory

### New Component: Password Change Form

**Location:** `apps/web/app/(app)/[workspaceSlug]/settings/page.tsx` (settings page) or `apps/web/components/password-change-modal.tsx` (modal variant)

**Placement decision:** Add a "비밀번호 변경" section within the existing settings page, or trigger a modal from the profile-edit-modal. The modal approach is recommended to match `profile-edit-modal.tsx` pattern.

### States

| State | Visual | Behavior |
|-------|--------|----------|
| **Default** | 3 password fields (current, new, confirm) + submit button | All fields empty, button enabled |
| **Validating** | Real-time hint pills below "new password" field | Show password strength checks as user types |
| **Field Error** | Red border on invalid field + error text below field | Triggered on submit with invalid input |
| **Submitting** | Button text changes to "변경 중...", button disabled (opacity 0.55) | Prevent double-submit |
| **Success** | Toast notification "비밀번호가 변경되었습니다" (type: success) | Modal closes, toast appears |
| **API Error (wrong password)** | Red alert box above form: "현재 비밀번호가 올바르지 않습니다." | Form stays open, fields preserved |
| **Account Locked** | Amber warning alert with countdown: "계정이 잠겼습니다. N분 후에 다시 시도해주세요." | Submit button disabled during lockout |
| **Rate Limited** | Amber warning alert: "시도가 너무 많습니다. 잠시 후 다시 시도해주세요." | Submit button disabled |

### Password Validation Hint Pills

Reuse `getPasswordChecks()` pattern from `reset-password/page.tsx`:
- "8자 이상" -- `password.length >= 8`
- "영문 포함" -- `/[a-zA-Z]/.test(password)`
- "숫자 포함" -- `/\d/.test(password)`
- "특수문자 포함" -- `/[!@#$%^&*()_+\-=[\]{};':"\\|,.<>/?]/.test(password)`

Pill style: `fontSize: 11px`, `padding: 2px 8px`, `borderRadius: 9999`, green when passed, gray when not.

## Interaction Contract

### Submit Flow

```
1. User fills all 3 fields
2. Client-side validation:
   a. Current password: required (non-empty)
   b. New password: >= 8 chars, contains letter + digit + special char
   c. Confirm: matches new password
3. If validation fails -> show field errors, do NOT submit
4. If validation passes -> set isSubmitting=true, disable button
5. PUT /api/v1/users/me/password { currentPassword, newPassword }
6. On success (200):
   a. Update auth tokens in memory (response includes new access + refresh token per D-03)
   b. Close modal
   c. Show success toast
7. On error:
   a. INVALID_CREDENTIALS -> red alert "현재 비밀번호가 올바르지 않습니다."
   b. ACCOUNT_LOCKED -> amber alert with countdown timer (reuse login lockout pattern)
   c. RATE_LIMITED -> amber alert
   d. INVALID_PASSWORD -> server message passthrough
   e. Network error -> generic error message
8. Set isSubmitting=false
```

### Keyboard & Accessibility

| Interaction | Behavior |
|-------------|----------|
| Tab order | Current password -> New password -> Confirm -> Cancel -> Submit |
| Enter key | Submits form (standard `<form>` behavior) |
| Escape key | Closes modal (if modal variant) |
| Focus visible | `2px solid var(--accent)` outline, `2px` offset (global rule) |
| Screen reader | `aria-label` on close button, `role="alert"` on error/warning alerts |
| Autocomplete | `current-password` on first field, `new-password` on second and third |
